## 對您的 Pull Request 的要求

- [ ] 不接受完全由 AI 生成的 Pull Request。
- [ ] 在您還在處理指南要求時，請勿開啟 Draft / WIP（草稿/進行中）狀態的 Pull Request。當您提交 Pull Request 時，它應該已經 100% 準備就緒並符合所有指南。**請改用 [#2242](https://github.com/sindresorhus/awesome/issues/2242) 以增加孵化可見性**。
- [ ] **請勿浪費我的時間。** 做好工作，遵守所有指南，並及時回應。
- [ ] **您必須審查至少 4 個其他的 [開放 Pull Request](https://github.com/sindresorhus/awesome/pulls?q=is%3Apr+is%3Aopen)。**
	嘗試優先處理未經審查的 PR，但您也可以在已審查的 PR 中添加更多評論。審查時請參考下表。此要求旨在幫助 Awesome 專案實現自給自足。請在此處評論您審查了哪些 PR。您應該在審查中投入足夠的精力並保持透徹。參考之前的 PR 審查以獲取靈感。**僅僅評論 "looks good" 或簡單地將 Pull Request 標記為已批准是不算數的！** 您必須實際指出錯誤或改進建議。指出 Lint 違規的評論是被允許的，但**不**計入正式審查。

	- ✅ `Add Software Architecture`
	- ❌ `Update readme.md`
	- ❌ `Add Awesome Swift`

	- ✅ `- [iOS](…) - 適用於 Apple 手機與平板的行動作業系統。`
	- ✅ `- [Framer](…) - 交互式 UI 設計的原型設計。`

- [ ] 條目的標題應使用標題大小寫 (Title Case)，且清單的 URL 應以 `#readme` 結尾。

## 對您的 Awesome 清單的要求

- [ ] **已存在至少 30 天。**<br>這意味著從第一次實際提交或開源（以最近者為準）起算 30 天。
- [ ] 不是由 AI 生成的。
- [ ] 對您的清單運行 [`awesome-lint`](https://github.com/sindresorhus/awesome-lint) 並修復報告的問題。如果有誤報或無法/不應修復的事項，請[報告](https://github.com/sindresorhus/awesome-lint/issues/new)。
- [ ] 預設分支名稱應為 [`main`，而非 `master`](https://www.zdnet.com/article/github-to-replace-master-with-alternative-term-to-avoid-slavery-references/)。
- [ ] **在 Readme 頂部包含對專案/主題的簡潔描述。** [(範例)](https://github.com/willempienaar/awesome-quantified-self)
	- ✅ `適用於 Apple 手機與平板的行動作業系統。`
	- ✅ `交互式 UI 設計的原型設計。`
	- ❌ `用於 iOS 開發的資源與工具。`
	- ❌ `Awesome Framer 套件與工具。`
- [ ] 這是辛勤工作的結果，也是我能產出的最好成果。
	**如果您沒有在清單上投入相當大的努力，您的 Pull Request 將立即被關閉。**
- [ ] 倉庫名稱應採用小寫 slug 格式：`awesome-name-of-list`。
	- ✅ `awesome-swift`
	- ✅ `awesome-web-typography`
	- ❌ `awesome-Swift`
	- ❌ `AwesomeWebTypography`
- [ ] 清單的標題（Heading）應採用 [標題大小寫 (Title Case)](https://capitalizemytitle.com/) 格式：`# Awesome Name of List`。
	- ✅ `# Awesome Swift`
	- ✅ `# Awesome Web Typography`
	- ❌ `# awesome-swift`
	- ❌ `# AwesomeSwift`
- [ ] GitHub 倉庫中的非生成 Markdown 文件。
- [ ] 倉庫應包含 `awesome-list` 與 `awesome` 作為 [GitHub 主題 (Topics)](https://docs.github.com/en/repositories/managing-your-repositorys-settings-and-features/customizing-your-repository/classifying-your-repository-with-topics)。我鼓勵您添加更多相關主題。
- [ ] 不是重複的。請搜尋現有的提交。
- [ ] 僅包含優秀的 (Awesome) 項目。Awesome 清單是對最好的東西的精選，而不是所有東西。
- [ ] 不包含未維護、已封存、已棄用或缺少文檔的項目。如果您確實需要包含此類項目，它們應放在單獨的 Markdown 文件中。
- [ ] 儘可能包含專案標誌/插圖。
	- 居中、全寬或放置在 Readme 的右上角。[(範例)](https://github.com/sindresorhus/awesome-electron)
	- 圖像應連結到專案網站或任何相關網站。
	- **圖像應為高 DPI (High-DPI)。** 將其設定為原始圖像寬度的一半。
	- 不要同時包含文字標題 `Awesome X` 與標誌 `Awesome X`。您可以將頁首圖像放在 `#` (Markdown 標題) 或 `<h1>` 中。
- [ ] 條目具有描述，除非標題本身已足夠具描述性。但這種情況很少見。
- [ ] 包含 [Awesome 徽章](https://github.com/sindresorhus/awesome/blob/main/awesome.md#awesome-badge)。
	- 應放置在 Readme 標題的右側。
		- 如果清單具有居中的圖形標題，則可以居中放置。
	- 應連結回本清單。
- [ ] 具有目錄 (Table of Contents) 章節。
	- 應命名為 `Contents`（目錄），而非 `Table of Contents`。
	- 應為清單中的第一個章節。
	- 應只有一級[嵌套列表](https://commonmark.org/help/tutorial/10-nestedLists.html)，最好沒有。
	- 不得包含 `Contributing`（貢獻）或 `Footnotes`（腳註）章節。
- [ ] 具有適當的授權。
	- **我們強烈建議使用 [CC0 授權](https://creativecommons.org/publicdomain/zero/1.0/)，但任何 [創用 CC (Creative Commons) 授權](https://creativecommons.org/choose/) 都可以。**
		- 提示：您可以通過訪問此 URL 快速將其添加到您的倉庫：`https://github.com/<用戶>/<倉庫>/community/license/new?branch=main&template=cc0-1.0`（相應替換 `<用戶>` 與 `<倉庫>`）。
	- MIT、BSD、Apache、GPL 等代碼授權是不被接受的。也不接受 WTFPL 與 [Unlicense](https://unlicense.org)。
	- 在倉庫根目錄放置一個名為 `license` 或 `LICENSE` 的文件，並附上授權文本。
	- **不要**在 Readme 中添加授權名稱、文本或 `Licence` 章節。GitHub 已經在倉庫頂部顯示了授權名稱與全文連結。
	- 為了驗證您已閱讀所有指南，請在您的 Pull Request 中僅評論單詞 `unicorn`。
- [ ] 具有[貢獻指南](https://github.com/sindresorhus/awesome/blob/main/awesome.md#include-contribution-guidelines)。
	- 文件應命名為 `contributing.md`。大小寫由您決定。
	- 它可以選擇從 Readme 中標題為 `Contributing` 的專用章節連結，該章節位於主要內容的頂部或底部。
	- 該章節不應出現在目錄中。
- [ ] 所有不重要但必要的內容（如額外的版權聲明、來源超連結、指向廣泛內容的指針等）應歸組在 Readme 底部的 `Footnotes`（腳註）章節中。該章節不應出現在目錄中。
- [ ] 具有一致的格式與正確的拼寫/語法。
	- 連結與描述之間用連字符分隔。<br>範例：`- [AVA](…) - JavaScript 測試執行器。`
	- 描述以大寫字母開頭，以句點結尾。
	- 一致且正確的命名。例如，`Node.js`，而非 `NodeJS` 或 `node.js`。
- [ ] 不使用 [硬換行 (Hard-wrapping)](https://stackoverflow.com/questions/319925/difference-between-hard-wrap-and-soft-wrap)。
- [ ] 不包含 CI（如 GitHub Actions）徽章。<br>您仍然可以使用 CI 進行 Lint，但徽章在 Readme 中沒有價值。
- [ ] 不在 Readme 頂部包含 `Inspired by awesome-foo` 或 `Inspired by the Awesome project` 之類的連結。Awesome 徽章就足夠了。
